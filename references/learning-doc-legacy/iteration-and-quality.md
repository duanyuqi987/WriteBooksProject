# 迭代、字数与质量检查

## 分批写作协议

大体量文档必须分批写作。不要承诺单轮写完 10 万字、30 万字或更多。

默认批次：

| 目标规模 | 每轮建议 |
|---|---|
| 1到3万字 | 每轮 1到2 章 |
| 5到10万字 | 每轮 1 章或 3到5 个小节 |
| 10到30万字 | 每轮 1 个大章或 8到15 个知识点 |
| 30万字以上 | 先拆分册，再逐册生成 |

每轮结束汇报：

```text
已完成：第X章到第Y节
估算有效正文：N 字
累计进度：N / 目标字数
本轮新增来源：source_id 列表
下一轮计划：...
```

如果用户已经明确“不用问我，持续写”，未达到目标前继续下一批。只有遇到版权风险、关键来源缺失、目标冲突或工具限制时停下说明。

## 字数估算

中英文混合文档不要只用空格拆词。推荐：

- 中文、日文、韩文字符按字符计入。
- 英文、数字和下划线连续片段按词计入。
- 去掉 HTML 标签、脚本、样式、目录模板和长代码块后估算正文。

可运行：

```bash
python3 scripts/count_text.py path/to/document.html
```

脚本输出 `estimated_total` 作为本项目的本地验收口径。它不是出版级精确字数，但用户指定目标字数时，必须达到或超过这个估算值才能宣称整本书达标。

## 目标字数硬约束

用户明确要求 `N 字`、`N 万字`、`不少于 N 字` 时，本项目把它作为硬性下限处理：

- 写作前先把目标换算成整数，例如 `10 万字 = 100000`。
- 每轮写完后运行 `python3 scripts/count_text.py <html文件>`。
- 只有当 `estimated_total >= 目标字数` 时，才可以说整本书已经达到用户字数要求。
- 如果 `estimated_total < 目标字数`，汇报时只能说“当前累计 N / 目标字数”，并继续下一批写作。
- 不得通过重复段落、目录、样式、脚本、长源码、网页原文或版权书原文凑字数。
- 如果用户要求“恰好 N 字”，仍以达到学习质量和不低于 N 字为优先；最终汇报实际估算量，说明这是项目脚本估算值。

## 来源追踪检查

每章至少能说清：

- 主要来源是什么。
- 哪些段落是来源转述。
- 哪些段落是原创桥接或教学设计。
- 哪些内容是推断。
- 哪些内容需要后续补源。

网站和在线文档要保留链接。代码库要保留文件路径。书籍要保留书名、章节或用户提供片段编号，避免长引原文。

## 质量检查清单

写完每轮后检查：

- 大纲顺序是否符合学习路径，而不是来源堆叠。
- 新术语首次出现时是否解释。
- 中英文术语是否一致。
- 每个知识点是否有例子、图示或练习。
- 源码讲解是否标注路径，并解释设计原因。
- 是否存在连续大段原文、源码或网页搬运。
- 是否存在“显然”“众所周知”等压过读者困惑的表达。
- 是否有复习题、常见坑和过渡句。
- HTML 的章节 ID 是否稳定。
- 修订日志是否更新。

## 事实与推断

正文中要区分事实、推断和教学化补充：

- **事实**：来自源码、官方文档、用户提供材料或可核验网页。
- **推断**：根据文件名、调用关系、配置或上下文判断，必须写明“从当前结构推断”。
- **教学化补充**：为帮助理解而加入的原创类比、例子、练习，不应伪装成来源原意。

## 更新旧文档

当用户要求“更新某章”“补上某节”“项目变了，重写对应部分”时：

1. 读取旧 HTML。若遇到历史遗留 Markdown，只把它当作来源材料转换进新的 HTML，不再作为最终书稿格式。
2. 定位稳定 ID。
3. 比对新来源和旧来源。
4. 只替换目标章节。
5. 更新目录、元信息、字数和修订日志。
6. 在汇报中说明未改动的章节。

不要因为局部更新而全文重写。全文重写会丢失历史修订、破坏链接，也更容易引入无关变动。

## 完成标准

可以宣称完成前，至少满足：

- 目标文档或本轮章节已经写入目标文件。
- 来源地图已覆盖用户给出的主要资料。
- 用户指定目标字数时，`estimated_total` 已达到或超过目标；阶段性续写则必须明确剩余字数。
- HTML 能打开，目录链接格式正确。
- 修订日志有本轮记录。
- 未解决问题明确列出。

如果不能验证，要明确说明未验证项、原因和风险。
