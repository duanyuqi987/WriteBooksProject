# 来源识别与处理

## 总原则

任何“根据资料写学习文档”的任务，都先建立来源地图。来源地图不是最终正文，而是写作前的事实底座，用来防止凭空发挥、漏掉重要材料、混淆版本或踩到版权边界。

## 来源地图字段

| 字段 | 说明 |
|---|---|
| source_id | 给每个来源分配稳定编号，如 `book-01`、`web-03`、`repo-core` |
| 类型 | 书籍、网站、代码库、项目文件夹、论文、说明文档、日志、配置、混合资料 |
| 语言 | 中文、英文、双语、代码/配置为主 |
| 主题 | 这个来源主要解释什么 |
| 可信度 | 官方、一手资料、项目内部资料、二手整理、未知 |
| 时效性 | 版本号、发布日期、最后更新时间、仓库提交时间 |
| 可引用范围 | 可短引、可转述、可链接、只可做结构参考 |
| 关键内容 | 目录、模块、概念、核心文件、页面栏目 |
| 缺口 | 读者要学会目标所需但来源里没有的内容 |
| 冲突 | 与其他来源不一致的术语、结论、版本或实现 |

## 中文书与英文书

处理书籍时，目标是“学习化再创作”，不是复刻原书。

1. 用户提供目录、摘录、笔记或本地文件时，先基于已给材料建立章节地图。
2. 用户只给书名时，不能假装读过全文。需要说明可基于公开信息、用户提供摘录和通用知识搭建学习框架；如需精确到章节细节，应让用户提供目录或片段。
3. 英文书先建立术语表：原文术语、推荐中文译法、第一次出现的章节、容易误译点。
4. 中文书保留作者自己的核心概念名，但正文要用原创解释重讲。
5. 不要输出大段连续原文，不要写成“替代购买或阅读原书”的章节级复刻。
6. 可做的内容包括：学习路线、概念拆解、原创例子、动手验证、代码实验、读书问题清单、章节关系图。只有用户明确要求教案时，才加入课堂活动和评分安排。

## 中文网站与英文网站

网站内容可能随时变化。涉及网站、在线文档、博客、产品说明、API 文档时，必须确认当前页面内容。

1. 优先读取官方文档、项目主页、仓库文档、标准规范或作者原文。
2. 记录链接、页面标题、发布日期或版本号、访问日期。
3. 英文网站不要机械全文翻译；应提炼概念、保留关键术语、补中文解释和实践例子。
4. 中文网站不要直接搬运段落；应重构为体系化学习路径。
5. 多个网页属于同一站点时，先整理站点栏目树，再决定章节顺序。
6. 如果网页包含当前价格、版本、法规、模型、接口、依赖版本等可能变化的信息，写作前必须重新核验。

### 大型英文技术网站抓取顺序

大型技术文档站不要只读首页，也不要盲目全站抓取。先抓“学习主线”：

1. 首页：确认项目定位、当前版本、发布日期、主导航。
2. 入门或 primer：提取背景知识和作者推荐学习顺序。
3. 安装页：记录依赖、平台差异、版本限制。
4. Tutorials 或 Getting Started：建立初学者路径。
5. API overview 或 architecture：识别核心对象模型。
6. 专题页：例如 planners、state spaces、benchmarking、Python bindings、FAQ。
7. 外部工具页：只记录链接和角色，不把外部站点当成已核验事实，除非另行打开核验。

写作时按学习目标重组，不按网站导航机械排列。对于英文原站，正文应使用中文原创讲解，保留关键英文术语和官方链接；引用原文只保留必要短句。

## 代码库

代码库优先从真实结构出发，不从框架经验倒推。

推荐扫描顺序：

1. `rg --files` 查看文件总览。
2. 读取 `README*`、`docs/`、`examples/`、`tests/`。
3. 识别依赖文件：`package.json`、`pyproject.toml`、`requirements.txt`、`Cargo.toml`、`go.mod`、`CMakeLists.txt`、`pom.xml`、`build.gradle`。
4. 找入口文件、路由、CLI、服务启动脚本、配置样例。
5. 找核心数据结构、领域模型、接口边界、错误处理和测试覆盖。
6. 建立模块关系：谁调用谁、数据如何流动、配置如何影响行为。

写源码导读时，不要把源码大段粘贴到文档里。每个代码片段应满足：

- 只截取说明问题所需的最小片段。
- 标明文件路径和必要行号。
- 讲清“为什么这么写”，而不是只翻译每行语法。
- 用图解释调用链、状态流、数据结构或生命周期。

## 项目文件夹与资料库

项目文件夹可能混有文档、代码、图片、表格、配置、导出文件和历史版本。先分类，再写作。

分类建议：

| 类别 | 常见文件 | 处理方式 |
|---|---|---|
| 入口说明 | README、使用手册、安装说明 | 作为读者起点和术语来源 |
| 代码实现 | `src/`、`lib/`、脚本 | 建立模块图和运行路径 |
| 配置样例 | yaml、json、toml、env | 解释参数含义和改动影响 |
| 数据样例 | csv、jsonl、sqlite、日志 | 用于设计实验和案例 |
| 设计资料 | md、pdf、docx、pptx | 提取概念、流程、架构图 |
| 历史沉淀 | old、backup、archive | 只在必要时引用，避免污染主线 |

若资料重复或版本冲突，优先级默认是：用户直接要求 > 当前项目说明 > 现行代码和配置 > 官方资料 > 历史文档 > 第三方文章。若这个顺序不适用，先说明新的判断标准。

## 混合素材

混合素材最容易写散。必须先做三件事：

1. 合并同义概念：例如 `Agent`、`智能体`、`代理` 是否指同一事物。
2. 标注冲突：例如书中理论和代码实现不一致，网站版本和本地依赖不同。
3. 设计桥接章节：让书籍理论、网站说明、项目代码、练习任务能按学习顺序衔接。

混合素材输出时，不按来源堆章节，而按学习目标组织章节。来源只作为证据和例子出现。

## 缺失与不可访问来源

遇到缺源时不要编造：

- 资料不存在：标为“待补充来源”，并继续完成不依赖该来源的部分。
- 网站无法访问：说明访问失败，改用用户提供内容或官方替代来源。
- 书籍未提供正文：只写原创学习框架、导读问题和通用背景，不写具体章节细节。
- 代码缺测试或入口：标出风险，用 README、配置和文件结构推断，并说明“这是推断”。

## 来源地图输出样例

```markdown
## 来源地图

| source_id | 类型 | 语言 | 主题 | 可信度 | 写作用途 |
|---|---|---|---|---|---|
| repo-core | 代码库 | 代码/英文 | 核心实现 | 项目一手资料 | 源码导读、实验 |
| web-official | 网站 | 英文 | 官方概念说明 | 官方 | 术语、版本说明 |
| notes-zh | 笔记 | 中文 | 用户经验 | 项目内部资料 | 常见坑、实战章节 |

缺口：缺少部署截图和真实运行日志。
冲突：官方文档使用新版参数名，本地代码仍使用旧参数名。
```
