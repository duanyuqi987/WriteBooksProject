# 文档类型蓝图

## 先选主形态

同一批资料可以写成不同书内主线。先判断用户真实目标，再决定这本 HTML 学习书采用哪种组织方式。注意：本项目最终产物统一是中文书名的单文件 HTML；下表不是输出格式分支，而是书稿内容的组织侧重。

| 用户目标 | 推荐书内主线 | 典型 HTML 书稿 |
|---|---|---|
| “教会我”“小白能懂” | 入门到进阶学习书 | 中文 HTML 学习书 |
| “给学生上课”“写教案” | 教师用书附录或授课版章节 | 中文 HTML 教师用学习书 |
| “整理成课程” | 课程化学习书 | 中文 HTML 课程学习书 |
| “像书一样”“写成书”“我是小白”“一个个理解算法和代码” | 学习书 | 长篇技术书、算法学习书、源码理解书 |
| “看懂这个项目/代码” | 源码导读学习书 | 架构图、调用链、模块讲解 |
| “把这个网站整理一下” | 网站知识学习书 | 栏目重组、概念索引、FAQ |
| “根据这本书学习” | 书籍学习指南 | 读书路线、概念图、练习 |
| “这些资料都整理成体系” | 混合素材学习书 | 多源融合专题书 |

## 学习文档

适合目标：自学、长期阅读、从入门到进阶。

推荐结构：

1. 学习地图：读者会学到什么、前置知识、建议阅读顺序。
2. 入门篇：用场景和问题引入，不先堆定义。
3. 基础篇：术语、背景、核心概念。
4. 实战篇：例子、项目、实验、配置、运行。
5. 深入篇：原理、设计取舍、边界条件。
6. 排障篇：常见错误、调试方法、检查清单。
7. 进阶篇：扩展方向、进一步阅读。
8. 附录：术语表、参考资料、FAQ、自查答案。

每章至少包含：章首导读、核心概念、算法或原理正文、例子或实验、图解、章末小结、读者自查、下一章过渡。不要把每一节都写成重复模板。

## 学习书

适合目标：用户明确说“当成书来写”“像书一样”“我是小白”“主要学习算法知识”“要一个个理解算法和代码”“多例子和图像”。

推荐结构：

1. 序章：这个领域解决什么问题，为什么值得学。
2. 基础篇：必要数学、术语、对象模型和第一个完整例子。
3. 算法篇：按算法家族讲解思想、步骤、图解、伪代码和手工推演。
4. 代码篇：安装、教程、API、Python 绑定、项目代码或示例代码逐段解释。
5. 进阶篇：最优、约束、多层、终止条件、Benchmark。
6. 实战篇：把算法放进真实项目、网站教程或本地代码库中运行和修改。
7. 附录：术语表、算法选择表、来源链接、FAQ、自查答案。

学习书正文应以“连续讲解”为主。每个算法专题建议使用：

- 一段真实场景引入。
- 一张算法图。
- 一段伪代码或最小代码。
- 一次手工推演。
- 一个数值例子。
- 一段代码逐行解释。
- 一组读者自查或动手验证。

禁止把学习书写成“卡片库”。如果 10 个连续小节都有完全相同的小标题和句式，说明写作失败。除非用户明确要教案，否则不要加入课时流程、授课动作、课堂环节或教师/学生角色分配。

## 教案

适合目标：老师授课、培训、内部分享、课程设计。

在本项目中，教案不能作为最终独立格式交付；如果用户要求教案，也写进中文 HTML 教师用书中，作为“授课建议、课时安排、练习与评价”章节或附录。

推荐结构：

1. 课程信息：主题、对象、时长、先修要求。
2. 教学目标：知识目标、能力目标、产出目标。
3. 重点难点：哪些必须掌握，哪些容易误解。
4. 教学准备：资料、环境、代码、网页、设备。
5. 时间安排：按分钟拆分导入、讲解、演示、练习、总结。
6. 教学过程：教师活动、学生活动、板书或投影内容。
7. 课堂练习：题目、步骤、预期结果、讲评点。
8. 课后作业：基础题、提高题、项目题。
9. 评价方式：自测题、评分标准、验收标准。
10. 教学反思：可调整点、补充材料、后续课衔接。

教案不是普通文章，要能被人直接拿去上课。每个环节都要有时间、动作和产出。

## 课程讲义

适合目标：培训材料、章节讲义、配套教材。

在本项目中，课程讲义作为中文 HTML 学习书的课程化组织方式，不作为 Markdown、PPT 或独立讲义文件交付。

推荐结构：

- 本讲目标
- 上讲回顾
- 新概念引入
- 正文讲解
- 示例和演示
- 课堂练习
- 知识检查
- 本讲小结
- 下讲预告

课程化章节比普通学习书更紧凑，比教师用书更面向学生阅读。

## 源码导读

适合目标：读懂代码库、项目交接、二次开发。

推荐结构：

1. 项目全景：这个项目解决什么问题。
2. 如何运行：环境、依赖、配置、启动路径。
3. 目录结构：每个顶层目录的责任。
4. 核心流程：请求流、数据流、控制流或任务流。
5. 模块导读：模块职责、关键类/函数、输入输出。
6. 关键设计：为什么这样拆分、有哪些取舍。
7. 修改指南：常见需求应该改哪里。
8. 测试与调试：如何验证修改没有破坏行为。
9. 常见坑：配置、依赖、环境、边界条件。

源码导读必须把文件路径、运行路径和概念讲解绑定起来。

## 网站知识整理

适合目标：把官方文档、博客、产品网站、教程站整理成体系。

推荐结构：

1. 站点地图：栏目、页面、版本、可信度。
2. 读者路线：先读哪些页面，哪些后读。
3. 概念索引：关键术语和中文解释。
4. 主题章节：按学习目标重组网页内容。
5. 操作指南：从网页信息落到实际步骤。
6. FAQ：常见问题和页面链接。
7. 更新风险：哪些页面需要后续复查。

不要按网页顺序机械搬运，要按学习顺序重组。

## 书籍学习指南

适合目标：辅助读书、拆解英文书、把书变成学习路线。

推荐结构：

1. 阅读前准备：这本书适合谁、要补什么背景。
2. 全书地图：主题、章节群、概念关系。
3. 分阶段阅读计划：每天或每周读什么、做什么练习。
4. 概念讲解：用原创语言解释关键思想。
5. 读书问题：每章带着哪些问题读。
6. 实践任务：把书中思想应用到项目或练习。
7. 术语表：尤其适合英文书。
8. 复盘模板：读完后如何总结和迁移。

不要写成原书替代品。重点是帮助用户读懂和使用这本书。

## 混合素材课程

适合目标：书、网站、代码、文件夹资料混在一起，需要整理成体系。

推荐结构：

1. 多源地图：每个来源负责什么。
2. 统一术语：不同来源的叫法如何对齐。
3. 学习路径：先理论、再工具、再项目，或先项目、再补理论。
4. 主题模块：每个模块引用多个来源互相支撑。
5. 案例项目：把资料中的知识落到真实任务。
6. 阶段测验：检查是否掌握。
7. 延伸阅读：按来源回链。

混合素材课程的章节标题应围绕“学习任务和理解路径”，不是围绕“来源名称”。

## 大体量估算

| 目标字数 | 建议知识点数 | 适合结构 |
|---|---:|---|
| 1到3万字 | 15到30个 | 快速入门、短课 |
| 5到10万字 | 50到100个 | 系统学习书、课程化学习书 |
| 10到30万字 | 120到300个 | 完整技术书、项目深度导读书 |
| 30万字以上 | 300个以上 | 分册 HTML 学习书 |

估算公式：`目标字数 ÷ 每个知识点平均字数 = 知识点数量`。如果用户要求极大字数，应优先拆成分册或阶段，不要把单章写得臃肿。
