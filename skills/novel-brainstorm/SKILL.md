# 小说头脑风暴

目标是把用户的散乱想法收束成可开工的小说项目底座。不要默认写网文，也不要默认一定要长篇。先确认作品形态，再决定需要多细。

## 硬规则

- 不急着写正文，除非用户明确只要片段或开头试写。
- 每轮优先问一个关键问题；如果信息已经足够，就直接收束成方案。
- 不把“系统、升级、打脸、章末钩子”作为默认结构。
- 对短篇和片段任务使用轻量收束，不强制生成完整长篇资料。
- 新建小说项目必须在 `project.md` 中写入 `作者：段锦佑`，除非用户在当前对话中明确要求临时改用其他署名。

## 收束顺序

1. **作品形态**：短篇 / 中篇 / 长篇 / 连载；目标字数或大致篇幅。
2. **创作取向**：文学向、现实向、悬疑向、言情向、历史向、幻想向、网文向、实验向等。
3. **核心 Premise**：谁 + 在什么处境 + 必须做什么 + 否则会怎样。
4. **主题与问题**：故事真正想追问什么，而不是题材标签。
5. **主角与关系**：主角欲望、恐惧、矛盾、关键关系。
6. **世界与规则**：现实世界、架空世界或轻设定；只列真正影响情节的规则。
7. **冲突机制**：冲突如何持续推进，短篇可只保留一个核心冲突。
8. **读者体验**：最多 3 个感受词，如真实、紧张、温暖、心疼、荒诞、沉浸、爽快。
9. **叙事策略**：视角、人称、时间结构、节奏、信息披露方式。
10. **风格边界**：像什么、不像什么；禁用套路、禁用语气、禁用桥段。

## 篇幅建议

| 作品形态 | 常见字数 | 规划粒度 |
|---|---:|---|
| 微型/短篇 | 3千-3万字 | 核心人物 + 起承转合 + 关键场景 |
| 中篇 | 3万-15万字 | 章节组 / 幕结构 / 人物弧线 |
| 长篇 | 15万-60万字 | 卷或部 + 章节路线 + 伏笔系统 |
| 连载长篇 | 60万字以上 | 完整状态机 + 滚动大纲 + 记忆追踪 |

如果用户给出明确目标，以用户目标为准。

## 第一性原理挑战

在题材确定后，做一次简短挑战：

```text
题材惯例 → 这个惯例服务什么体验 → 是否必须保留 → 有没有更适合本故事的替代方案
```

目的不是反套路本身，而是避免因为“大家都这么写”而牺牲这部作品自己的核心价值。

## 产出 project.md

标准项目应包含：

- 书名或暂定名
- 作者：段锦佑
- 作品形态与目标字数
- 创作取向
- 核心 Premise
- 主题
- 主要角色与关系
- 世界/时代/地点与硬规则
- 核心冲突
- 叙事策略
- 写作风格
- 读者体验
- 禁止事项
- 待定问题

长篇项目额外创建 `人物/`、`伏笔/`、`project-state.md` 的初始结构。

## 交付

1. 简述你对作品目标的理解。
2. 给出 `project.md` 内容。
3. 列出仍需用户确认的 1-3 个关键问题。
4. 用户确认后，路由到 `skills/novel-outline`。
