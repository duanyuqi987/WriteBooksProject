# 长文写作系统

## 写作目标

学习书和学习文档的目标不是“把资料变长”，而是让读者真的能学会。好的长文应该做到：

- 新概念出现前有铺垫。
- 抽象概念后面有例子。
- 代码片段后面有设计解释。
- 每章都能让读者知道自己是否真正理解。
- 章节之间有清晰过渡。

## 学习书的基本写法

当用户要求“像书一样”“写成书”“我是小白”“一个个理解算法和代码”“学习算法知识”时，必须写成连续的技术书正文。它不是表格化教案，也不是卡片库。每章应该像真正的书一样，有引入、有铺垫、有推演、有代码、有回望。

推荐的章内节奏：

1. **场景钩子**：用一个真实问题、操作场景或困惑开头。
2. **类比解释**：用日常事物解释直觉，但不要让类比替代定义。
3. **正式定义**：给出准确表述、边界和适用范围。
4. **图示**：用 SVG、Mermaid、ASCII 或表格解释关系。
5. **公式或伪代码**：把直觉落成可执行的算法语言。
6. **数值例子或手工推演**：用小地图、小状态空间、小代码把算法跑一遍。
7. **代码对应**：只截关键片段，讲每行为什么这样写、输入输出是什么、改错会影响哪里。
8. **边界提醒**：说明适用条件、失败现象和排查方向。
9. **小结过渡**：总结本节，并自然引到下一节。

学习书正文禁止把连续小节都写成同一套固定标题，例如反复出现“目标卡片、提示卡片、活动卡片、问题卡片”。这些内容可以转化为自然正文、旁注、图注、读者自查或章末回顾。

## 算法知识写法

算法型资料要优先讲“算法怎么工作”，而不是只讲工具怎么调用。

每个核心算法至少覆盖：

- 问题定义：输入、输出、约束、成功标准。
- 算法步骤：可写成自然语言、编号步骤或伪代码。
- 直观图像：树、图、搜索边界、投影网格、代价收敛曲线等。
- 小例子：用 2D 平面、迷宫、机械臂关节或状态点手工推演。
- 适用场景：什么时候推荐，什么时候不推荐。
- 与相邻算法比较：例如 PRM vs RRT，RRTConnect vs RRT*，KPIECE vs EST。
- 工程参数：如采样数、连接半径、range、projection、termination condition、time limit。
- 失败模式：窄通道、无解问题、高维空间、错误状态空间、碰撞检查过粗。
- 代码含义：核心类、函数、变量和参数如何对应算法步骤。
- 读者路径：小白先看直觉和例子，有基础读者再看公式、伪代码和工程边界。

## 受众校准

| 读者级别 | 写法重点 | 避免 |
|---|---|---|
| 小白 | 先讲为什么，再讲是什么，最后讲怎么做 | 一开头堆术语、公式、源码 |
| 有基础 | 快速补齐背景，增加对比和实践 | 重复过多常识 |
| 进阶 | 强调取舍、边界、源码细节和失败案例 | 只给入门解释 |
| 工程师 | 给文件路径、命令、配置影响和调试路径 | 空泛概念，不落到项目 |

如果用户明确要教案或授课材料，再补课时安排、讲授动作和作业设计。仅仅说“给老师和学习者参考”不等于要把正文写成授课流程。

## 中英文资料处理

英文资料写中文文档时，第一次出现术语使用：

```text
中文译名（English Term, 缩写）：一句自然中文解释。
```

示例：

```text
依赖注入（Dependency Injection, DI）：把对象需要的依赖从外部传进来，而不是在对象内部自己创建。
```

规则：

- 专有名词不要强行意译到失真。
- 同一个术语全篇保持同一译法。
- 对容易混淆的词建立术语表，如 `agent`、`workflow`、`pipeline`、`runtime`。
- 引用英文原句必须短，并立即解释它的含义和影响。
- 不要把英文网站整页翻译成中文，要重构成学习路径。

## 代码讲解比例

代码库或项目文档中，建议保持“代码 1：讲解 4到6”的比例。

每段代码讲解应回答：

1. 这段代码在项目流程中的位置是什么？
2. 它接收什么输入，产生什么输出？
3. 为什么这里要这样设计？
4. 改错这里会影响哪些调用方？
5. 初学者最可能误解哪一行？

不要为了凑字数贴完整文件。只截取最小必要片段。

## 图示规则

优先使用能帮助理解的图，而不是装饰图。

常见图型：

| 场景 | 推荐图 |
|---|---|
| 概念关系 | 概念图、树形图 |
| 代码调用 | 调用链、时序图 |
| 数据处理 | 流程图、管道图 |
| 系统架构 | 模块图、分层图 |
| 学习路径 | 路线图、阶段表 |
| 对比选择 | 决策表 |

Mermaid 单图节点建议不超过 15 个。复杂图拆成多张，先给全景，再给局部。

## 自查与动手验证

长篇学习书可以加入自查和动手验证，但不要让它盖过书籍正文。常见形式：

1. **读者自查**：用 2 到 4 个问题确认是否理解本章。
2. **动手验证**：运行一个最小例子、改一个参数、观察一张图。
3. **迁移思考**：换一个机器人、状态空间或项目场景重新套用同一思想。

教案另有自己的评分标准和课堂流程，不应混入学习书的每一节。

## 阶段回顾

每 2到4 章插入一次阶段回顾：

- 这几章解决了什么问题。
- 哪些概念已经铺好。
- 哪些问题还没展开。
- 接下来为什么自然进入下一阶段。

阶段回顾能防止长文变成资料堆积。

## 禁忌

- 不要使用“众所周知”“显然”“很简单”压过读者困惑。
- 不要连续多段只有抽象概念，没有例子、图或练习。
- 不要把“参考资料顺序”误当成“学习顺序”。
- 不要在没有来源支撑时写成确定事实。
- 不要用“作为 AI”之类表达进入正文。
- 不要为了显得专业堆英文术语。
