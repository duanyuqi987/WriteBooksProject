# 增量更新与版本管理

## HTML 元信息

每份可迭代 HTML 都建议包含：

```html
<meta name="doc-version" content="v1.0">
<meta name="last-update" content="2026-05-13 12:00:00">
<meta name="word-count" content="100000">
<meta name="source-policy" content="summarized-and-transformative">
```

## 修订日志

文档末尾保留：

```html
<section id="changelog">
  <h2>修订日志</h2>
  <ul>
    <li><b>v1.2 (2026-05-13 15:30:00)</b>：补充第3章，新增英文官方文档来源。</li>
    <li><b>v1.1 (2026-05-13 11:20:00)</b>：更新第2章源码导读，修正配置说明。</li>
    <li><b>v1.0 (2026-05-12 20:10:00)</b>：首次生成。</li>
  </ul>
</section>
```

## 稳定 ID

| 层级 | ID |
|---|---|
| 章 | `chapter-1` |
| 节 | `sec-1-2` |
| 小节 | `subsec-1-2-3` |
| 知识点 | `point-1-2-3-4` |

更新文档时不要随意改 ID。ID 是目录、链接和局部替换的锚点。

## 局部更新流程

1. 读取现有文档。
2. 定位要更新的章节 ID。
3. 读取新来源并更新来源地图。
4. 只替换目标 `<section>`。
5. 更新目录中的标题或状态。
6. 更新字数、时间和版本号。
7. 在修订日志顶部追加一条。
8. 汇报哪些章节被修改，哪些没有动。

## 版本号建议

- `v1.0`：首次可用版本。
- `v1.1`：小范围章节补充或修正。
- `v2.0`：大纲结构、目标读者或主要来源发生重大变化。

## 不要做的事

- 不要为更新一章而全文重写。
- 不要删除旧修订日志。
- 不要把未验证的新来源写成确定事实。
- 不要更新目录却忘了正文 ID。
- 不要改动用户明确要求保留的章节。
